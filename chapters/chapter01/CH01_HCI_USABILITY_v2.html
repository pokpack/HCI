Usability Engineering
& ISO 9241-11

วัดได้ · เชิงวิทยาศาสตร์ · มาตรฐานสากล

เชื่อมจาก UCD: เราออกแบบโดยยึดผู้ใช้ แต่จะรู้ได้อย่างไรว่า "ดีพอ"? ต้องวัด — Usability Engineering คือวิธีวัด ISO 9241-11 คือมาตรฐานที่บอกว่าต้องวัดอะไร

Usability Engineering = วิธีการเชิงวิทยาศาสตร์ในการออกแบบและวัด Usability ต่างจาก UX ที่กว้างกว่า Usability คือ subset ของ UX ที่วัดได้ชัดเจนด้วย metrics เชิงปริมาณ

Usability กับ UX ต่างกันอย่างไร? LO4

Usability
  • วัดได้ด้วย metrics
  • Objective + Subjective
  • Task-based measurement
  • ISO 9241-11 standard

UX (User Experience)

  • รวม Emotion, Delight, Trust

  • Holistic experience

Usability ⊂ UX — ในรายวิชานี้เน้น Usability Engineering เพราะวัดได้และ engineer สามารถควบคุมได้
ความแตกต่างสำคัญ: Usability วัดได้ด้วยตัวเลข UX กว้างกว่าและ subjective มากกว่า ในฐานะ engineer เราเน้น Usability Engineering เพราะมี metrics ที่ชัดเจน วัดซ้ำได้ และ actionable

Effectiveness — ทำสำเร็จหรือเปล่า? LO6

Task Completion Rate = (Tasks สำเร็จ ÷ Tasks ทั้งหมด) × 100%
ตัวอย่าง
ผู้ทดสอบ 10 คน ทำ task "ซื้อสินค้า"
สำเร็จ 8 คน | ล้มเหลว 2 คน
8 ÷ 10 × 100% = 80%
Benchmark
≥ 85% — Excellent
70–84% — Good
50–69% — Needs Improvement
< 50% — Poor, Redesign Required
🙋 ถามสด: 10 คนทดสอบ สำเร็จ 7 คน = ?%
ถามสด: 10 คนทดสอบ สำเร็จ 7 คน = 70% → Needs Improvement สำคัญ: ต้องกำหนด success criteria ก่อนทดสอบ ไม่ใช่หลัง เช่น "task สำเร็จ = กดยืนยันการชำระเงินสำเร็จ" ไม่ใช่ "พยายามแล้วแต่ไม่ได้"

Efficiency — ใช้เวลา/resource เท่าไหร่? LO6

Avg. Time on Task = Σ Time ÷ จำนวนผู้สำเร็จ task
Efficiency ครอบคลุม:
  • ⏱ เวลาเฉลี่ยต่อ task
  • 🖱 จำนวน click/action
  • ❌ จำนวน error
  • 🔄 จำนวน backtrack
Benchmark: Time on Task
< 60 วินาที — Efficient ✅
60–120 วินาที — Acceptable ⚠

SUS คืออะไร และวัดอะไร? LO8

System Usability Scale
  • สร้างโดย John Brooke (1986)
  • 10 คำถาม Likert 1–5
  • Score 0–100
  • Free + Technology-Agnostic
  • Validated ทั่วโลก
⚠ SUS วัดอะไร?
วัด Perceived Usability — ความรู้สึกของผู้ใช้
ไม่ใช่ Functionality · ไม่ใช่ Features · ไม่ใช่ "ดีกว่าคู่แข่ง"
SUS เป็น free, validated, technology-agnostic tool ใช้ได้กับทุก system John Brooke สร้างขณะทำงานที่ Digital Equipment Corporation 1986 และ adapt มาจาก "quick and dirty" usability test ปัจจุบันเป็นมาตรฐานที่ cite มากที่สุดในงานวิจัย HCI

🎯 Positive/Negative สลับกัน — ป้องกัน Acquiescence Bias (ตอบ "เห็นด้วย" ทุกข้อโดยไม่คิด)
อ่านออกเสียงร่วมกัน ให้สังเกต positive/negative สลับกันเพื่อกัน acquiescence bias (การตอบเห็นด้วยทุกข้อโดยอัตโนมัติ) — ข้อคี่ = positive (ยิ่งมาก = ยิ่งดี) ข้อคู่ = negative (ยิ่งมาก = ยิ่งแย่)

ปุ่ม "ยืนยันการชำระเงิน" มีสีเทา เหมือนปุ่ม disabled ทำให้ผู้ใช้ 4/10 คนไม่กดต่อ → CR Task 3 = 60%
Severity: Level 4 (Critical)
After (Recommendation):
เปลี่ยนสีปุ่มยืนยันเป็น #0F4C81 เข้ม + เพิ่ม label "คลิกเพื่อชำระ" → คาดว่า CR จะ ≥ 85%
Severity ช่วยจัดลำดับความสำคัญในการแก้ไข วิศวกรต้องระบุว่าอะไรต้องแก้ก่อน Before/After format ทำให้ Recommendation ชัดเจนและ actionable ไม่ใช่แค่บอกว่า "ปรับปรุง UI"

  • 5. Metrics — CR, Time, ER, SUS

Process Flow: Usability Engineering ใน 5 ขั้น NEW

1
Define Context
ผู้ใช้คือใคร ใช้ที่ไหน ทำงานอะไร ข้อจำกัดคืออะไร
2
Set Metrics
กำหนด CR, Time, Error Rate, SUS ก่อนเริ่มทดสอบ
3
Build Prototype
ทำหน้าจอที่พอให้ผู้ใช้ทำ task จริงได้
4
Test With Users
สังเกต พูดให้น้อย วัดผลให้ครบ และจด evidence
5
Report & Improve
สรุป finding, severity, recommendation แล้ว iterate
